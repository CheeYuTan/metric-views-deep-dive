Title:
Databricks Metric Views Deep Dive Part 3: Window Semantics for Time-Aware Business Metrics

Subtitle:
How Metric Views define YTD, rolling, prior-year, leading, inclusive/exclusive, and semiadditive metrics in the semantic layer.

Core takeaway:
Window semantics let a Metric View own time intelligence so dashboard authors do not repeat fragile window SQL.

Prerequisites and caveats:
This article uses YAML 1.1 Metric Views and advanced window measure features. Window measures are documented as experimental. The `offset` modifier and inclusive/exclusive trailing or leading ranges require newer runtime support. Check the current Databricks feature availability page before using these patterns in production.

The problem:
Finance dashboards are full of time-aware metrics:

- Current month revenue
- Year-to-date revenue
- Running total revenue
- Rolling 12-month revenue
- Same month prior year
- YoY growth
- Quarter-end balance

These metrics are easy to get wrong when every dashboard writes its own SQL.

Metric Views let us define those semantics once.

Window mental model:

```mermaid
flowchart TB
  ORDER["order field<br/>fiscal_month"]
  RANGE["range<br/>current / cumulative / trailing / leading / all"]
  SEMI["semiadditive<br/>last or first when order field is absent"]
  OFFSET["offset<br/>shift frame for prior period"]
  MEASURE["window measure"]

  ORDER --> MEASURE
  RANGE --> MEASURE
  SEMI --> MEASURE
  OFFSET --> MEASURE
```

Current month:

The simplest window is current.

measures:
  - name: current_month_revenue
    expr: SUM(amount) FILTER (
      WHERE source_grain = 'GL'
        AND scenario_id = 'ACTUAL'
        AND account_category = 'Revenue'
    )
    window:
      - order: fiscal_month
        range: current
        semiadditive: last

This says:

- Calculate revenue at the fiscal month grain.
- If the query groups by fiscal month, return that month.
- If the query does not group by fiscal month, use semiadditive behavior.

Running total:

measures:
  - name: running_total_revenue
    expr: SUM(amount) FILTER (
      WHERE source_grain = 'GL'
        AND scenario_id = 'ACTUAL'
        AND account_category = 'Revenue'
    )
    window:
      - order: fiscal_month
        range: cumulative
        semiadditive: last

This gives all revenue up to the current month.

YTD:

YTD is a cumulative measure with a reset boundary.

measures:
  - name: ytd_revenue
    expr: SUM(amount) FILTER (
      WHERE source_grain = 'GL'
        AND scenario_id = 'ACTUAL'
        AND account_category = 'Revenue'
    )
    window:
      - order: fiscal_month
        range: cumulative
        semiadditive: last
      - order: fiscal_year_start
        range: current
        semiadditive: last

The first window accumulates over fiscal month.

The second window restricts the cumulative calculation to the current fiscal year.

Rolling 12:

measures:
  - name: rolling_12_month_revenue
    expr: SUM(amount) FILTER (
      WHERE source_grain = 'GL'
        AND scenario_id = 'ACTUAL'
        AND account_category = 'Revenue'
    )
    window:
      - order: fiscal_month
        range: trailing 12 month inclusive
        semiadditive: last

The word inclusive matters.

It means the anchor month is included in the 12-month frame.

Inclusive versus exclusive:

Metric Views support both:

- trailing 3 month exclusive
- trailing 3 month inclusive

Example:

measures:
  - name: trailing_3_month_revenue_exclusive
    expr: SUM(amount) FILTER (...)
    window:
      - order: fiscal_month
        range: trailing 3 month exclusive
        semiadditive: last

  - name: trailing_3_month_revenue_inclusive
    expr: SUM(amount) FILTER (...)
    window:
      - order: fiscal_month
        range: trailing 3 month inclusive
        semiadditive: last

Exclusive excludes the anchor month.

Inclusive includes the anchor month.

Prior year with offset:

Offset shifts the window frame.

measures:
  - name: prior_year_revenue
    expr: SUM(amount) FILTER (
      WHERE source_grain = 'GL'
        AND scenario_id = 'ACTUAL'
        AND account_category = 'Revenue'
    )
    window:
      - order: fiscal_month
        range: current
        semiadditive: last
        offset: -12 month

This gives the same fiscal month one year earlier.

Then compose YoY:

measures:
  - name: yoy_revenue_growth
    expr: MEASURE(current_month_revenue) - MEASURE(prior_year_revenue)

  - name: yoy_revenue_growth_pct
    expr: MEASURE(yoy_revenue_growth) / NULLIF(MEASURE(prior_year_revenue), 0)

This is a powerful pattern:

Define shifted windows first, then compose growth metrics from them.

Leading windows:

Leading windows look forward.

measures:
  - name: next_month_revenue
    expr: SUM(amount) FILTER (...)
    window:
      - order: fiscal_month
        range: leading 1 month
        semiadditive: first

This is useful for forecast comparisons, pipeline views, or next-period projections.

Semiadditive balances:

Balances are the most important finance example.

You can sum balances across entities or products at a point in time.

You should not sum balances across months.

measures:
  - name: month_end_balance
    expr: SUM(balance_amount) FILTER (WHERE source_grain = 'BALANCE')
    window:
      - order: fiscal_month
        range: current
        semiadditive: last

This means:

- At month grain, return the month-end balance.
- At quarter or year grain, return the last available month value.
- Across entities/products/accounts, still aggregate normally.

Date hierarchy gotcha:

For window measures, hierarchy fields should be derived from the order field.

Good:

fields:
  - name: event_date
    expr: event_date
  - name: fiscal_month
    expr: DATE_TRUNC('MONTH', event_date)
  - name: fiscal_year_start
    expr: make_date(year(event_date), 1, 1)

Risky:

fields:
  - name: fiscal_month
    expr: raw_month_column

Even if two expressions look equivalent, defining hierarchy levels from the order field keeps the relationship clear for window aggregation.

Demo query:

SELECT
  fiscal_month,
  region,
  MEASURE(current_month_revenue) AS current_month_revenue,
  MEASURE(running_total_revenue) AS running_total_revenue,
  MEASURE(ytd_revenue) AS ytd_revenue,
  MEASURE(rolling_12_month_revenue) AS rolling_12_month_revenue,
  MEASURE(next_month_revenue) AS next_month_revenue,
  MEASURE(prior_year_revenue) AS prior_year_revenue,
  MEASURE(yoy_revenue_growth_pct) AS yoy_revenue_growth_pct
FROM finance_metric_view
WHERE fiscal_month >= DATE'2025-01-01'
GROUP BY ALL
ORDER BY fiscal_month, region;

Semiadditive query:

SELECT
  fiscal_quarter,
  entity_name,
  account_category,
  MEASURE(month_end_balance) AS month_end_balance
FROM finance_metric_view
WHERE statement_section = 'Balance Sheet'
  AND fiscal_year = 2025
GROUP BY ALL;

Design guidance:

- Keep atomic time measures clear.
- Compose growth metrics from window measures.
- Use month offsets for monthly data, not day approximations.
- Be explicit about inclusive versus exclusive windows.
- For balances, always ask: should this sum across time?
- Add comments to semiadditive measures because business users often misunderstand them.

Closing:
Window semantics make Metric Views much more than a governed aggregation layer.

They let the semantic model own time intelligence.

That means fewer duplicated dashboard calculations, fewer inconsistent YTD definitions, and fewer accidental balance rollups.
